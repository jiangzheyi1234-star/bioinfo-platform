  "content": "# Adding reference databases to Snakemake and nf‑core (Nextflow) pipelines — recommendation, schema, validation, UX and implementation patterns\n\nExecutive summary\n- For nf‑core–style Nextflow pipelines and for modular Snakemake workflows, a tool‑template‑first model (user selects a tool-specific template/record for each database entry, e.g., Kraken2, MetaPhlAn, CARD/RGI, or custom) is recommended when pipelines run tool‑specific modules and when the pipeline authors want precise, automated validation and strong reproducibility guarantees. This pattern is already used in nf‑core/taxprofiler (databases CSV with per‑row tool, db_name, db_params, db_path) and matches the way many nf‑core modules expect typed inputs [3], [6].  \n- That said, a simple generic/free‑form databases sheet (single schema covering many DBs) is workable for minimal onboarding or for pipelines that treat DBs as opaque filesystem assets; several nf‑core pipelines accept such CSVs but historically have had limited non‑file parameter validation (nf‑core/taxprofiler examples) [3], [4].  \n- Best practice is to combine the tool‑template approach with schema validation (JSON Schema for Snakemake; nf‑schema / samplesheetToList for nf‑core) and to provide clear guidance and a “custom” template path for user‑built databases. Use provenance capture (URL, version/tag, build command, checksum where possible), allow DBs as local or cloud assets (S3/GS/http) or refgenie URIs, and support both automated download/build flows and prebuilt‑DB usage as mutually explicit modes. The remainder of this report explains the tradeoffs, required/recommended metadata, concrete validation strategies and rules, integration patterns for Snakemake and nf‑core, real examples in the community, UX flows, deliverables (schema + examples), and evidence gaps.\n\nContents\n- Comparison: tool‑template‑first vs generic database sheet\n- Required and recommended metadata fields and extensibility\n- Concrete schema and validation strategies (Snakemake + nf‑core)\n- Integration patterns for provenance, DB management, and reproducibility\n- Survey of community practice and canonical examples\n- UX and developer validation flows and messaging\n- Concrete deliverables: canonical config schema (YAML/JSON + TSV example), example JSON Schema, Snakemake / nf‑core integration snippets, reproducibility checklist\n- Evidence gaps\n- References\n\n## Comparison: tool‑template‑first model (A) vs generic/free‑form database sheet model (B)\n\nOverview of observed practice\n- nf‑core/taxprofiler uses a databases CSV with defined columns (tool, db_name, db_params, db_path) that maps DB rows to tool runs and downstream modules; the pipeline does not bundle databases and requires users to supply them [3]. nf‑core pipelines also provide parameter validation and conversion utilities via the nf‑schema / samplesheetToList approach in Nextflow training and tooling [4], [5]. Snakemake supports JSON Schema validation of configuration and tabular sheets (snakemake.utils.validate) and can model tabular rows as records in a schema [1].\n\nAdvantages of tool‑template‑first (A)\n- Enables precise, tool‑specific validation: modules can assert presence/format of expected files (e.g., Kraken2 directories, MetaPhlAn bowtie2 index + marker pickle, CARD unarchived directory) so runtime errors are detected early instead of at classification time [6], [8], [13].\n- Better mapping to pipeline modules and options: per‑tool db_params fields (as used by taxprofiler) let pipeline code pass correct CLI flags or module inputs automatically [3].\n- Improves reproducibility: per‑tool metadata fields make recording provenance (tool name, DB name, version/tag, build command) structured and therefore easier to capture in logs and MultiQC style reports [3], [16].\n- Facilitates automation of DB build/download: template semantics allow the pipeline to conditionally run a builder (e.g., createtaxdb or tool‑specific build steps) when a matching template is selected [12].\n- Easier UI/UX for onboarding: users select a known tool template and follow a short checklist (required fields) rather than crafting free‑form entries.\n\nDisadvantages of tool‑template‑first (A)\n- More upfront maintenance cost for pipeline authors: templates/validators must be implemented and kept current for each supported tool (Kraken2, MetaPhlAn, CARD/RGI, etc.) and for their evolving filelayout/requirements [6], [8], [13].\n- Potentially higher learning curve when many tool types are supported; needs good documentation and “custom” flow for unusual databases.\n\nAdvantages of generic/free‑form sheet (B)\n- Low barrier to use: a single CSV/TSV format is easy to create and reuse across pipelines.\n- Minimal implementation burden for pipeline authors; pipeline treats DBs as opaque directories/files and documents required file paths instead of enforcing them.\n\nDisadvantages of generic/free‑form sheet (B)\n- Weak validation and higher runtime error rates: unless the pipeline performs additional checks, missing expected files or wrong formats are only discovered when the tool runs (examples: taxprofiler historically does not validate non‑file DB parameters) [3].\n- Harder to capture structured provenance automatically; free‑form fields are less likely to contain machine‑readable version/tag/build info.\n- Harder to automate DB builds/downloads because the sheet lacks a canonical “type” that maps to builders.\n\nSynthesis / recommendation\n- Adopt a tool‑template‑first schema as the primary, recommended experience for pipelines that call tool‑specific modules (nf‑core and modular Snakemake). Provide a clear “custom” template and a fallback generic record type for opaque or unsupported DBs. Couple templates with machine‑enforced schema validation at pipeline startup (Nextflow/nf‑schema or Snakemake JSON schema) so that mistakes are surfaced early [4], [1], [5]. For quick prototyping or minimal pipelines, allow a generic sheet but display clear warnings that less validation / reproducibility capture will occur.\n\n## Required and recommended metadata fields and formats\n\nPrinciples used to choose fields\n- Capture what is necessary to reproduce an analysis: source (URL), version/tag, build command and parameters, exact file paths used by the pipeline, and any tool‑specific inventory that the tool expects (e.g., Kraken2 index files). This aligns with community recommendations to record computational pipeline details, software versions, and processing parameters [16].\n\nCanonical fields (required vs recommended)\n- Required (minimum useful set for reproducibility and pipeline operation):\n  - tool (string): classifier/consumer tool identifier (e.g., kraken2, metaphlan, rgi, custom). This is how the pipeline maps rows to modules [3], [6].\n  - db_name (string): user label for the DB, unique within the sheet [3].\n  - db_path (string/URI): path or URI to the database on the local filesystem or cloud storage (support file://, s3://, gs://, http(s) and refgenie:// style URIs). Many nf‑core pipelines accept paths and archives as inputs; some default to S3 iGenomes paths [15], [3], [9].\n- Recommended (strongly encouraged):\n  - source_url (string): canonical download page, repository, or upstream source; used for provenance capture.\n  - version/tag (string): release tag, version string, or date indicating which upstream snapshot was used.\n  - build_command (string): the command(s) used to build the DB (if user built it), e.g., kraken2-build commands; helpful for reproducibility and re‑build automation [3].\n  - db_params (string/object): tool‑specific runtime flags or builder flags (taxprofiler uses a db_params field) [3].\n  - format (string): canonical format name (e.g., kraken2, metaphlan_bowtie2, card_rgi, fasta_indexed) to allow format‑specific validation.\n  - path_expected_files (array of strings): basename(s) the pipeline expects in that db_path (e.g., Kraken2: hash.k2d opts.k2d taxo.k2d; MetaPhlAn: markers .pkl + bowtie2 index files) [7], [6], [8].\n  - license (string): license or reuse constraints (e.g., RGI/CARD license conditions) [14].\n  - organism_scope / taxa_scope (string or controlled vocabulary): taxonomic scope or organism(s) included; useful when comparing DBs.\n  - note / description (string): human readable description.\n- Strongly recommended for machine verification but not present in all community configs (see Evidence Gaps below):\n  - checksum / digest (string): checksum of an archive or of key files to detect tampering or corruption (evidence does not specify standard practice; see Evidence Gaps).\n  - size_bytes (integer): size on disk or archive size (helpful for resource planning).\n  - build_date / release_date (string): timestamp.\n- Tool / format specific metadata (examples observed):\n  - Kraken2: indicate presence of hash.k2d, opts.k2d, taxo.k2d and whether taxonomy was downloaded via kraken2‑build --download-taxonomy [7].\n  - MetaPhlAn: bowtie2 index directory path and marker pickle (.pkl) presence; MetaPhlAn can auto‑download latest unless index specified [8].\n  - CARD / RGI: version of CARD used and whether the CARD download has been unarchived (nf‑core RGI module expects unarchived CARD directory) [13], [14].\n- Extensible custom DB mechanism:\n  - Provide a well‑documented “custom” tool value with required file checks declared by the user (path_expected_files, format, build_command) and optional structured fields. This allows pipelines to validate user‑built DBs even when they do not match a known tool template.\n\nSuggested config sheet formats\n- Human‑friendly tabular form (CSV/TSV) for users (nf‑core taxprofiler uses CSV columns: tool, db_name, db_params, db_path) [3]. This is convenient for editing and reuse between runs.\n- Machine‑friendly structured form (YAML or JSON): one object per DB entry with typed fields (for Snakemake JSON Schema validation and for Snakemake configfile usage) [1].\n- Provide both: a canonical YAML/JSON schema plus a TSV example derived from the same schema (deliverables below).\n\n## Concrete schema‑validation strategies and tools\n\nSupported/observed mechanisms in the ecosystem\n- Snakemake: provides snakemake.utils.validate which can validate a loaded config (dict, Pandas DataFrame, Polars DataFrame) against a JSON schema. Schemas can be JSON or YAML; the JSON Schema defaults property can populate missing entries [1].\n- Nextflow / nf‑core: validation and conversion utilities exist in the nf‑schema / samplesheetToList tooling; samplesheetToList reads CSV/TSV/JSON/YAML, validates them against a JSON schema, returns a Groovy list, and throws on validation failure. nf‑core pipelines enable parameter validation by default via params.validate_params = true and use UTILS_NFSCHEMA_PLUGIN for initialization checks [4], [5]. The nf‑validation spec documents sample sheet conventions and the behavior that fields present but not described in the schema are ignored with a warning [17].\n\nConcrete validation rule categories (implementable with JSON Schema + small content checks)\n- Structural / type checks (JSON Schema):\n  - Required fields exist and have the correct types (e.g., tool: string; db_path: string/URI; db_name: string).\n  - Allowed enumeration for tool values (kraken2, metaphlan, rgi, custom, …) where pipelines support only a fixed set.\n  - Default population for optional fields (JSON Schema defaults) so that missing optional metadata does not break downstream code [1].\n- Tabular / per‑row modeling:\n  - When validating CSV/TSV, model the expected record in the JSON Schema so each row is validated as a single JSON object (Snakemake supports validating DataFrame rows this way) [1].\n- File existence and simple file format checks (pipeline runtime preflight, not JSON Schema):\n  - Assert db_path exists (local) or is resolvable (HTTP/S3) before job submission. For directories: check presence of required files (e.g., Kraken2: hash.k2d, opts.k2d, taxo.k2d) [7].\n  - For compressed archives: assert supported compression suffix (e.g., .tar.gz) if pipeline can auto‑decompress.\n- Tool‑specific content verification (lightweight):\n  - Kraken2: check presence of at least hash.k2d, opts.k2d, taxo.k2d to consider it a Kraken2 DB [7].\n  - MetaPhlAn: check for marker pickle file (.pkl) and Bowtie2 indexfiles in specified bowtie2db directory [8].\n  - RGI/CARD: check that the supplied path contains the required CARD files or unarchived folder expected by the nf‑core RGI module [13], [14].\n- Remote resolution tests:\n  - For HTTP/S3 URIs, attempt a HEAD request or list operation to confirm existence and detect 4xx/5xx responses before starting heavy downloads (the nf‑core iGenomes base defaults to an S3 URI pattern in practice) [15].\n- Checksum validation (recommended but not specified by sources): include checksum field and, if provided, verify after download. Evidence does not provide a community standard for checksums (see Evidence Gaps).\n\nHow to combine JSON Schema + lightweight runtime checks\n- Use JSON Schema (Snakemake or nf‑schema) to perform all structural/type checks and to populate defaults [1], [4], [5]. Then run a preflight script or rule that implements filesystem/HTTP/S3 checks and tool‑specific file presence checks (using minimal I/O such as HEAD or list and existence checks of a few small files). This two‑stage approach is how nf‑core and Snakemake validation tooling is typically used in practice [4], [1].\n\n## Integrating database selection and validation into Snakemake and nf‑core pipelines\n\nHigh‑level patterns (compatible with evidence)\n- Validate at pipeline startup (preflight):\n  - For Nextflow (nf‑core): rely on params.validate_params and samplesheetToList (nf‑schema plugin) to reject bad DB sheets early; throw clear errors to prevent long runs with misconfigured DBs [4], [5].\n  - For Snakemake: call snakemake.utils.validate on the loaded config or DataFrame and abort on errors; use defaults to provide sane fallbacks [1].\n- Manage DBs as assets:\n  - Allow DBs to be supplied as local paths, cloud URIs (S3/GS/HTTP), or refgenie URIs; refgenie can be used to resolve and populate paths into a config file (refgenie populate replaces refgenie:// URIs with concrete paths) and refgenie maintains a config for assets [9], [11].\n- Two supported modes for DB acquisition:\n  - Prebuilt DBs supplied by user: Pipeline validates paths and expected files; optionally auto‑decompress archives if pipeline supports it (taxprofiler decompresses/extracts compressed DB archives) [3].\n  - Automated download/build: Pipeline maps a tool template to a builder script (e.g., kraken2‑build commands as tutorialed by taxprofiler) and either runs the builder or emits a reproducible build log and artifact. nf‑core/createtaxdb shows how builder pipelines can be implemented from CSV input to produce classifier DBs [12], [3].\n- Capture provenance in outputs and logs:\n  - Record and include the pipeline version used (nf‑core/taxprofiler logs pipeline version in MultiQC), and record per‑DB metadata (source_url, version/tag, build_command) in run logs and in an outputs/assets manifest for future reproducibility [3], [16].\n- Coordinate environment capture:\n  - Ensure that DB builders and consumers run in captured environments (conda/bioconda, mamba, or container images). nf‑core pipelines use profiles and required templates so users can pin execution environment and pipeline version together; pipeline version should be recorded in outputs [5], [3].\n- Handling archive formats and directory inputs:\n  - Accept both pre‑extracted directories and compressed archives and document pipeline behavior (e.g., mag accepted directory inputs of pre‑uncompressed archives in a release change) [15]. When auto‑unpacking is performed by the pipeline, log the exact unpack commands and archive checksum when available.\n\nConcrete integration snippets (conceptual)\n- Snakemake (validate config and load DB list):\n  - Use configfile: \"config/config.yaml\" and then call snakemake.utils.validate(config, schema) early in the workflow to check the DB records and populate defaults; implement a rule that checks path existence and required filenames [1].\n- Nextflow / nf‑core:\n  - Use samplesheetToList(params.databases, schema=json_schema) in pipeline init; set params.validate_params = true and fail fast if validation errors occur [4], [5]. Map each validated record to process inputs with conditional logic per tool.\n\n## Survey of existing implementations and community practices (examples)\n\nnf‑core/taxprofiler\n- Uses a databases CSV with columns tool, db_name, db_params, db_path. It requires users to supply DBs (no DBs included) and logs pipeline version in MultiQC for reproducibility; the pipeline historically did not validate non‑file DB parameters automatically in older versions but has evolved to validate sample sheets in nf‑core style workflows [3].\n\nnf‑core modules expectations\n- The nf‑core Kraken2 module expects a db parameter that is a directory representing a Kraken2 DB and outputs classification files; MetaPhlAn modules expect a directory containing the pre‑downloaded MetaPhlAn database (bowtie2 index dir + marker pickle) [6], [8]. The nf‑core RGI module expects an unarchived CARD download directory [13].\n\nTool documentation and builder examples\n- Kraken2 manual documents required files for a DB and commands to build/download taxonomy and libraries; a Kraken2 DB directory must contain at least hash.k2d, opts.k2d, and taxo.k2d [7]. Taxprofiler tutorials document kraken2 build steps and commands [3].\n- MetaPhlAn tooling can automatically download a database unless user specifies a custom index; MetaPhlAn requires a Bowtie2 db directory and a marker pickle file [8].\n- Refgenie provides a configuration store and commands to build assets and to populate refgenie:// URIs into concrete paths in templates/configs; refgenie build supports building derived assets by pulling parents when necessary [9], [10], [11].\n- nf‑core/createtaxdb is a canonical example of a pipeline that builds classifier DBs from CSV definitions and can generate downstream nf‑core samplesheets [12].\n\nSnakemake / config validation\n- Snakemake supports JSON Schema validation of YAML/JSON config and DataFrames and provides defaults to populate missing entries; Snakemake best practices recommend using config files and optional tabular sample sheets rather than runtime parameters [1], [2].\n\nReleases / operational examples\n- Several nf‑core pipelines accept iGenomes base S3 path style defaults for references and permit toggling iGenomes loading; some pipelines auto‑decompress provided archives (taxprofiler) while others expect prebuilt directories [15], [3].\n\n## UX and developer‑facing validation flows and messaging\n\nPreferred flows (supported by observed behavior)\n- Preflight / dry‑run validation at startup: use JSON Schema validation (Snakemake) or samplesheetToList (nf‑core) to fail fast and present clear field‑level errors to users instead of cryptic tool errors later [1], [4], [5].\n- Clear, actionable error messages: when a required file is missing, say which path and which required filename(s) were expected (e.g., “Kraken2 database missing hash.k2d in /path/to/db — run kraken2-build --build or supply a prebuilt DB”) — this aligns with the approach of enforcing existence checks for module inputs in nf‑core modules [6], [13].\n- Non‑interactive validation by default with an interactive mode option: nf‑core pipelines enable params.validate_params = true by default to perform non‑interactive validation; pipelines may offer a verbose or --fix flag to attempt automated fixes or to provide build suggestions [4]. If an automated builder is available (createtaxdb, refgenie support), present the exact build command as a suggested fix [12], [10].\n- Fallback behavior: if optional metadata is missing, use JSON Schema defaults to populate safe defaults (Snakemake supports defaults) and warn the user; require minimal fields for execution and mark advanced provenance fields (source_url, checksum) as recommended but optional [1].\n- Onboarding guidance for custom DBs: provide an explicit “custom” template with a short checklist (path_expected_files, format, build_command) and example commands (e.g., kraken2-build sequence of commands), and offer a small validation rule set that checks the claim (e.g., that path_expected_files are present) [3], [7], [12].\n\nConcrete messaging patterns\n- Preflight summary: list validated DB entries, indicating (OK) or an explicit error with remediation steps per failing entry.\n- If auto‑download/build is available: present a “Would you like pipeline to download/build the DB? [y/N]” with the exact command(s) that will be run shown (record these commands in the provenance manifest).\n- If DB is an archive and pipeline auto‑decompresses: show the target unpack path and log the exact unpack operation performed [3].\n\n## Concrete deliverables (ready to implement)\n\n1) Recommended canonical config schema (YAML/JSON conceptual layout; TSV example below)\n- Canonical YAML/JSON style (one DB entry per record):\n  - tool: string (enum: kraken2 | metaphlan | rgi | card | refgenie | custom | …)\n  - db_name: string\n  - db_path: string (path or URI; local dir or s3:// / gs:// / http(s) / refgenie://)\n  - db_params: string or object (tool runtime flags or builder flags)\n  - format: string (optional; e.g., kraken2, metaphlan_bowtie2)\n  - source_url: string (optional)\n  - version: string (optional)\n  - build_command: string (optional)\n  - path_expected_files: array[string] (optional; basenames the pipeline should find)\n  - license: string (optional)\n  - organism_scope: string (optional)\n  - description: string (optional)\n\n2) Human‑readable TSV example (small)\ntool\\t db_name\\t db_path\\t db_params\\t version\\t source_url\nkraken2\\t standard_kraken2\\t /data/db/kraken2/standard\\t --quick\\t 2020-01-01\\t ftp://ftp.ncbi.nlm.nih.gov/...\nmetaphlan\\t metaphlan_v20\\t /data/db/metaphlan/v20\\t --index mpa_v20_m200\\t v20\\t https://bitbucket.org/biobakery/metaphlan/src/\nrgi\\t CARD_2023\\t /data/db/CARD/2023\\t \\t 4.0.1\\t https://card.mcmaster.ca/\n\n(Above format mirrors taxprofiler CSV columns and extends them with optional provenance fields) [3], [8], [14].\n\n3) Example JSON Schema fragment (illustrative, suitable for Snakemake / nf‑schema)\n- A minimal JSON Schema to validate one DB record (adapt to pipeline needs; Snakemake and nf‑schema accept JSON/YAML schemas). Map each record to processes based on record.tool value [4], [5].\n\n6) Checklist of reproducibility guarantees (what to record and how)\n- Pipeline version and profile used (record in MultiQC / run manifest) — nf‑core pipelines record pipeline version in MultiQC [3].\n- Per‑database: tool, db_name, source_url, version/tag, build_command (or builder pipeline reference), db_path used in run, and whether the DB was supplied prebuilt or built by pipeline [3], [16].\n- Recording of environment used for builders/consumers (container image tag or conda env spec) — Snakemake envvars directive and nf‑core profiles capture environment choices [1], [5].\n- If using refgenie URIs, include the refgenie config entry used and the resolved local paths (refgenie populate can replace URIs in template files) [11], [9].\n- (Recommended) checksum or digest of the DB archive or key files to detect undetected changes — evidence does not specify a standard but adding and recording checksums is recommended (see Evidence Gaps).\n\n7) Quick UX recommendations\n- Fail fast on schema errors; provide a “dry‑run –validate” mode that lists DB entries and their validation status.\n- Provide automated builder suggestions: when an entry fails because files are missing but builder commands are known (template maps to builder), print the exact builder command and offer to run it (interactive or with a --build-databases flag).\n- For custom DBs, provide an interactive checklist or a small helper script that collects required metadata and produces a validated entry.\n- Always record provenance (see checklist) and include a per‑run DB manifest in results.\n\n## Evidence gaps\n- Checksums and digest practices: the reviewed sources do not specify a community standard for including or validating checksums for database archives or assets (no source provides a canonical checksum field practice). Explicit checksum verification is recommended but the evidence does not document an accepted format or tool for this in nf‑core / Snakemake pipelines [evidence does not specify].\n- Concrete examples of JSON Schema rules covering all database formats and checksum validation across nf‑core: the material shows that JSON Schema and nf‑schema are used to validate sample sheets, but there are few canonical cross‑tool schema examples for DB records beyond the taxprofiler CSV shape [4], [1], [3].\n- Prebuilt automated content validation depth: the evidence documents lightweight file presence checks (e.g., Kraken2 required .k2d files) but does not present standardized automated content validators (e.g., checking FASTA header taxonomy formats) for multiple DB types [evidence does not specify].\n\n## Conclusion\n- For robust reproducibility and low runtime error rates in Snakemake and nf‑core pipelines, prefer the tool‑template‑first model with explicit schema validation and a clear custom DB path. Use Snakemake’s JSON Schema support and nf‑core’s samplesheetToList / nf‑schema approaches to validate structural correctness and to fail fast. Complement schema validation with lightweight runtime checks for file existence and small tool‑specific file inventories (Kraken2 .k2d files, MetaPhlAn .pkl and Bowtie2 indexes, RGI/CARD directories). Capture provenance (pipeline version, source URL, version/tag, build command) in output manifests; consider integrating refgenie for asset resolution where appropriate. Where automatic building is supported, present the build commands and record them in the manifest so DB build is reproducible.
